# Release Hub テスト仕様書

## 1. 目的

Release Hubの主要機能、データ互換性、API永続化、デプロイ設定が仕様どおりであることを確認する。本書は現行の自動テストと、リリース前に実施する手動受入試験を定義する。

## 2. テスト方針

- データ形状・API・移行は自動テストを優先する。
- 時刻ドラッグ、レスポンシブ、モーダルなど視覚・操作性は実ブラウザでも確認する。
- テストデータは一時ディレクトリに作成し、本番相当データを変更しない。
- 変更範囲に応じて型チェック、production build、Nodeテストを一括実行する。
- GitHub Pages変更時はworkflow成功と公開URLを確認する。

## 3. テスト環境

| 項目 | 条件 |
| --- | --- |
| Node.js | 20以上。CIは22 |
| パッケージ導入 | `npm ci --ignore-scripts` |
| 自動テスト | Node組み込みtest runner |
| APIテスト | `PORT=0`、一時 `DATA_DIR` |
| ブラウザ試験 | モダンブラウザ、デスクトップ／モバイル幅 |
| デモ試験 | `VITE_DEMO_MODE=true`、`VITE_BASE_PATH=/release-hub/` |

## 4. 実行コマンド

```bash
npm test
```

`npm test` は以下を順に実行する。

1. `npm run typecheck`
2. `npm run build`
3. `node --test tests/release-hub.test.mjs`

## 10. CI/CD受入基準

### GitLab CI

- testステージが型チェックとNodeテストに成功する。
- buildステージがdistを生成する。
- 指定成果物が7日間保持される。

### GitHub Actions / Pages

- `main` pushのworkflowが成功する。
- build jobで `npm test` が成功する。
- `/release-hub/` をbase pathとして資産が読み込める。
- Pages URLでサンプル作業が表示される。
- デモで編集後、再読み込みすると初期状態へ戻る。

## 11. リリース判定

以下をすべて満たした場合にリリース可能とする。

- `npm test` が全件成功。
- 変更対象機能の受入試験が成功。
- データ形状変更時に移行試験が成功。
- 永続データ、シード、サンプル、型、API、文書が整合。
- GitHub Pages対象変更時にActionsが成功。
- 重大度High以上の既知不具合がない。

## 12. テスト証跡

リリース時は以下を記録する。

- 対象コミットSHA
- `npm test` の成功結果
- 実施した手動試験IDと結果
- GitHub ActionsまたはGitLab PipelineのURL
- 未実施項目と理由
- 既知の制約・残存リスク
